# StockFlow – Mini Inventory Management System

## Software Requirement Specification

## 1. Project Overview

StockFlow is a mini inventory management system designed for small businesses to manage products, stock-in transactions, stock-out transactions, and inventory reports.

The purpose of this project is to build a simple but realistic web application that can be used for manual testing, API testing, database testing, and basic automation testing practice.

## 2. Project Goals

The system should allow users to:

* Log in to the system.
* Manage product information.
* Record stock-in transactions.
* Record stock-out transactions.
* View inventory quantity.
* View inventory transaction history.
* Generate simple inventory reports.

This project is designed for a testing portfolio, so the system should have clear business rules, validation rules, and enough features to create meaningful test cases.

## 3. User Roles

### 3.1 Admin

Admin can:

* Log in.
* View dashboard.
* Create, update, delete, and view products.
* View stock-in transactions.
* View stock-out transactions.
* View inventory reports.
* Manage product status.

### 3.2 Staff

Staff can:

* Log in.
* View product list.
* Create stock-in transactions.
* Create stock-out transactions.
* View inventory reports.
* View transaction history.

Staff cannot:

* Delete products.
* Change product status.
* Access user management.

## 4. Functional Requirements

## 4.1 Authentication

### FR-AUTH-001: Login

The system shall allow users to log in using username and password.

Input fields:

* Username
* Password

Validation rules:

* Username is required.
* Password is required.
* Username must exist in the system.
* Password must match the stored password.

Expected behavior:

* If login is successful, the system redirects the user to the dashboard.
* If login fails, the system displays an error message.
* The system should not reveal whether the username or password is incorrect separately.

Error message:

```text
Invalid username or password.
```

### FR-AUTH-002: Logout

The system shall allow logged-in users to log out.

Expected behavior:

* After logout, the user is redirected to the login page.
* The user cannot access protected pages without logging in again.

## 4.2 Dashboard

### FR-DASH-001: View Dashboard Summary

The system shall display a dashboard summary after login.

Dashboard should show:

* Total products
* Total active products
* Total inactive products
* Total stock quantity
* Number of low-stock products
* Number of out-of-stock products

Business rules:

* Low-stock product means quantity is greater than 0 and less than or equal to 10.
* Out-of-stock product means quantity equals 0.

## 4.3 Product Management

### FR-PROD-001: View Product List

The system shall allow users to view the product list.

Product list should display:

* Product ID
* Product name
* Category
* Price
* Quantity
* Status
* Created date
* Updated date

The system should support:

* Search by product name.
* Filter by category.
* Filter by status.
* Sort by created date.

### FR-PROD-002: Create Product

The system shall allow Admin to create a new product.

Input fields:

* Product name
* Category
* Price
* Initial quantity
* Status

Validation rules:

* Product name is required.
* Product name must be unique.
* Category is required.
* Price is required.
* Price must be greater than 0.
* Initial quantity is required.
* Initial quantity must be greater than or equal to 0.
* Status is required.

Default values:

* Status default value is Active.
* Initial quantity default value is 0.

Expected behavior:

* If product creation is successful, the product appears in the product list.
* If validation fails, the system displays the corresponding validation message.
* The system must not create duplicate products with the same name.

### FR-PROD-003: Update Product

The system shall allow Admin to update product information.

Editable fields:

* Product name
* Category
* Price
* Status

Validation rules:

* Product name is required.
* Product name must be unique.
* Category is required.
* Price must be greater than 0.
* Status must be Active or Inactive.

Business rules:

* Product quantity cannot be edited directly from the product update screen.
* Quantity can only be changed through stock-in or stock-out transactions.

### FR-PROD-004: Delete Product

The system shall allow Admin to delete a product.

Business rules:

* A product can be deleted only if its quantity is 0.
* A product cannot be deleted if it has stock-in or stock-out transaction history.
* If deletion is not allowed, the system displays an error message.

Error message:

```text
Cannot delete product because it has inventory quantity or transaction history.
```

### FR-PROD-005: Change Product Status

The system shall allow Admin to change product status.

Status values:

* Active
* Inactive

Business rules:

* Inactive products cannot be used in new stock-in transactions.
* Inactive products cannot be used in new stock-out transactions.
* Inactive products are still visible in reports and transaction history.

## 4.4 Stock-In Management

### FR-STIN-001: Create Stock-In Transaction

The system shall allow Staff and Admin to create a stock-in transaction.

Input fields:

* Product
* Quantity
* Note

Validation rules:

* Product is required.
* Product must exist.
* Product must be Active.
* Quantity is required.
* Quantity must be greater than 0.
* Note is optional.

Expected behavior:

* When stock-in is successful, the product quantity increases.
* A stock-in transaction record is created.
* The transaction must include created date and created by user.

Example:

Current product quantity: 20
Stock-in quantity: 5
New product quantity: 25

### FR-STIN-002: View Stock-In History

The system shall allow users to view stock-in transaction history.

Transaction history should display:

* Transaction ID
* Product name
* Quantity
* Note
* Created by
* Created date

The system should support:

* Search by product name.
* Filter by date range.

## 4.5 Stock-Out Management

### FR-STOUT-001: Create Stock-Out Transaction

The system shall allow Staff and Admin to create a stock-out transaction.

Input fields:

* Product
* Quantity
* Reason

Validation rules:

* Product is required.
* Product must exist.
* Product must be Active.
* Quantity is required.
* Quantity must be greater than 0.
* Quantity must not exceed current product quantity.
* Reason is optional.

Expected behavior:

* When stock-out is successful, the product quantity decreases.
* A stock-out transaction record is created.
* The transaction must include created date and created by user.

Example:

Current product quantity: 20
Stock-out quantity: 5
New product quantity: 15

Error case:

Current product quantity: 3
Stock-out quantity: 5
Expected result: The system rejects the transaction.

Error message:

```text
Stock-out quantity cannot exceed current inventory quantity.
```

### FR-STOUT-002: View Stock-Out History

The system shall allow users to view stock-out transaction history.

Transaction history should display:

* Transaction ID
* Product name
* Quantity
* Reason
* Created by
* Created date

The system should support:

* Search by product name.
* Filter by date range.

## 4.6 Inventory Report

### FR-REP-001: View Inventory Report

The system shall allow users to view inventory reports.

Report should display:

* Product name
* Category
* Current quantity
* Stock status
* Last updated date

Stock status rules:

* In stock: quantity greater than 10.
* Low stock: quantity greater than 0 and less than or equal to 10.
* Out of stock: quantity equals 0.

The system should support:

* Filter by category.
* Filter by stock status.
* Export is out of scope for version 1.

## 5. Non-Functional Requirements

## 5.1 Usability

* The UI should be simple and easy to use.
* Validation messages should be clear.
* Main actions should have confirmation messages.

## 5.2 Performance

* Product list should load within 3 seconds for 1,000 records.
* Search and filter should return results within 3 seconds.

## 5.3 Security

* Users must log in before accessing the system.
* Passwords must not be stored as plain text.
* Staff users must not access Admin-only actions.
* Protected APIs must require authentication.

## 5.4 Data Integrity

* Product quantity must always be accurate after stock-in and stock-out.
* Stock-out must never make product quantity negative.
* Transaction history must not be deleted when product status changes.

## 6. Out of Scope

The following features are not included in version 1:

* User registration
* Password reset
* Supplier management
* Customer management
* Purchase order management
* Sales order management
* Barcode scanning
* Export Excel/PDF
* Email notification
* Multi-warehouse management
* Payment management

## 7. Main Business Rules

| Rule ID | Business Rule                                                               |
| ------- | --------------------------------------------------------------------------- |
| BR-001  | Product name must be unique.                                                |
| BR-002  | Product price must be greater than 0.                                       |
| BR-003  | Product quantity cannot be negative.                                        |
| BR-004  | Product quantity can only be changed through stock-in or stock-out.         |
| BR-005  | Stock-in quantity must be greater than 0.                                   |
| BR-006  | Stock-out quantity must be greater than 0.                                  |
| BR-007  | Stock-out quantity cannot exceed current inventory quantity.                |
| BR-008  | Inactive products cannot be used in new stock-in or stock-out transactions. |
| BR-009  | Products with quantity greater than 0 cannot be deleted.                    |
| BR-010  | Products with transaction history cannot be deleted.                        |
| BR-011  | Low stock means quantity is from 1 to 10.                                   |
| BR-012  | Out of stock means quantity equals 0.                                       |

## 8. Main Modules

| Module             | Description                                           |
| ------------------ | ----------------------------------------------------- |
| Authentication     | Login and logout                                      |
| Dashboard          | Summary of product and inventory status               |
| Product Management | Create, view, update, delete, search, filter products |
| Stock-In           | Create and view stock-in transactions                 |
| Stock-Out          | Create and view stock-out transactions                |
| Inventory Report   | View current inventory status                         |

## 9. Acceptance Criteria

The project is considered complete when:

* Users can log in and log out.
* Admin can create, update, delete, and view products.
* Staff can view products but cannot delete products.
* Users can create stock-in transactions.
* Users can create stock-out transactions.
* Product quantity updates correctly after stock-in and stock-out.
* The system prevents stock-out quantity from exceeding current inventory.
* The system prevents negative quantity.
* The system shows inventory report correctly.
* API endpoints work correctly in Postman.
* Database records are updated correctly.
* Manual test cases can be executed and recorded.
